# Security

## Credential storage

formseal-fetch stores credentials (API tokens, namespace IDs) in your operating system's secure credential storage:

| OS | Storage location |
|---|------------------|
| Windows | Credential Manager |
| macOS | Keychain |
| Linux | DBus Secret Service |

### OS keychain

This is the preferred and secure method:
- Credentials are encrypted at rest by the OS
- Access requires your user account
- Managed by the OS — leverages built-in security features

Use `fsf status` to verify:

```
Token Location:           OS Keychain
```

### Fallback behavior

If the OS keychain is unavailable (e.g. headless container, missing keyring service), credentials fall back to `secrets.json` in the config directory:

```
~/.config/formseal-fetch/secrets.json
```

This file is base64-encoded but not encrypted. Any process with access to this file can read the credentials.

When the fallback is active, `fsf status` shows:

```
Token Location:           Config File
```

This is not an error — credentials still work — but OS keychain is strongly recommended.

## What gets stored

| Data | Stored as | Location |
|------|-----------|----------|
| API token | Encrypted | OS Keychain (preferred) or secrets.json |
| Namespace ID / project ref | Plaintext | config.json or secrets.json |
| Key prefix | Plaintext | config.json |
| Provider name | Plaintext | config.json |
| Output folder | Plaintext | config.json |

## What is NOT stored

- Private keys (handled by [formseal-decrypt](https://github.com/useFormseal/decrypt))
- Plaintext submissions
- Decrypted data

## Encryption guarantee

Ciphertexts are encrypted at the source by [formseal-embed](https://github.com/useFormseal/embed) using X25519 sealed boxes. fsf only handles encrypted data — a backend compromise yields no plaintext.

## Security guarantees

formseal-fetch:

- **Never sends your API token anywhere** — only to your chosen storage backend
- **Never sends data to third parties** — only communicates with your configured backend
- **No telemetry or analytics** — no hidden network calls during operation
- **Stores credentials securely** — uses OS keychain when available

## What to protect

### API tokens

If your API tokens are exposed, an attacker can access your storage backend and download ciphertexts.

**Best practices:**
- Restrict token permissions to the minimum required (Read-only for KV, etc.)
- Rotate tokens periodically in your provider dashboard
- Reconnect with `fsf disconnect` when done on shared machines
- Never commit tokens to version control

### Ciphertext files

Downloaded ciphertexts (`formseal.ct.jsonl`) are encrypted but still sensitive:
- Store in a secure location
- Decrypt promptly and delete raw ciphertexts when no longer needed
- Use `fsf disconnect --wipe` to remove them on disconnect

## What formseal-fetch does NOT protect against

- **Compromised storage backend** — ciphertexts are encrypted, but the backend can deny access or delete data
- **Credential theft from the local machine** — OS keychain mitigates this, but a compromised user account can still access stored credentials
- **Malware on the system** — an attacker with local access can intercept tokens in memory during operation
- **Physical access to the machine** — encrypted keychain helps, but a dedicated attacker with physical access may extract secrets

## Threat model

formseal-fetch is a local CLI tool in a pipeline. It assumes:

- The storage backend is trusted to store ciphertexts (but not to read them)
- The local machine is trusted by the user
- The user account is not compromised

It is designed for:

```
formseal-embed encrypts (browser)
    → backend stores ciphertext
    → fsf fetches ciphertext
    → fsd decrypts locally
```

Each stage has separate credentials and separate privileges.

## Clearing credentials

```bash
fsf disconnect
```

You'll be prompted to confirm (`y/N`) before anything is deleted.

This deletes:
- API token from OS Keychain
- Provider-specific credentials
- Configuration file (`config.json`)
- Fallback secrets file (if exists)

Downloaded ciphertext files are **NOT** affected by a plain `disconnect`.

To also delete ciphertext files:

```bash
fsf disconnect --wipe
```

`--wipe` removes the ciphertexts file from your output folder.

## Verifying keychain storage

Run `fsf status` — if "Token Location" shows "OS Keychain", your credentials are stored securely.
